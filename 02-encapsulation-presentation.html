OOP Timeline: 1960s

Simula — The Pioneer of OOP

Developed by Ole-Johan Dahl and Kristen Nygaard in Norway.

Introduced objects, classes, and instances, laying the groundwork for OOP.

[1960s] Simula: the first OO language

  • Two Norwegians — Kristen Nygaard and Ole-Johan Dahl — working for the Norwegian Defense research
  • In 1960s they created Simula to solve modeling problems
  • They were working on simulations that deal with exploding ships, and realized they could group the ships into different categories
  • Each ship type would have its own class, and the class would generate its unique behavior and data
  • Simula was not only responsible for introducing the concept of a class, but it also introduced the instance of a class

Dahl & Nygaard

Ole-Johan Dahl

Ole-Johan Dahl was a Norwegian computer scientist. Dahl was a professor of computer science at the University of Oslo and is considered to be one of the fathers of Simula and object-oriented programming along with Kristen Nygaard.

Turing Award 2001

Kristen Nygaard

Kristen Nygaard was a Norwegian computer scientist, programming language pioneer, and politician. Internationally, Nygaard is acknowledged as the co-inventor of object-oriented programming and the programming language Simula with Ole-Johan Dahl in the 1960s.

Turing Award 2001

OOP Timeline: 1970s — 1980s

Smalltalk — Bringing OOP to the Mainstream

Development at Xerox PARC by Alan Kay and others.

Featured a fully object-oriented environment and was instrumental in introducing the term "object-oriented programming."

[1970s] Smalltalk: the purist OO language

  • The term OOP was first used by Alan Kay at Xerox PARC in their Smalltalk language
  • The term was used to refer to the process of using objects as the foundation for computation
  • The Smalltalk team was inspired by the Simula project, but they designed Smalltalk so that it would be dynamic
  • Purist approach:
    • Everything is an object
    • No primitive types (int, float...)
    • Message-passing style, not method-invocation style

OOP Timeline: 1980s — 1990s

The Rise of C++ and Java

C++ added object-oriented features to C, becoming widely used for system/software development.

Java, designed with the network in mind, emphasized portability and security, making OOP principles more accessible and popular.

1980s: Commercial & Academic Adoption

  • Throughout the 1980s, OOP gained popularity in both academic and commercial settings
  • Languages like Objective-C (developed by Brad Cox and Tom Love) introduced objects to the C language
  • C++ (originally named "C with Classes") was developed by Bjarne Stroustrup, adding object-oriented features to the C language and gaining widespread use in the software industry
  • The concept of OOP also spread to other languages, including Eiffel (developed by Bertrand Meyer), which introduced design by contract, and Ada, which included OOP features in its 1983 version
Objective-C C++ Eiffel Ada

1990s: Mainstream Adoption & Java

  • The 1990s saw the mainstream adoption of OOP, most notably with the introduction of Java by Sun Microsystems in 1995
  • Designed with a focus on portability, security, and simplicity, Java's "write once, run anywhere" philosophy contributed to its widespread adoption for web and enterprise development
  • The Unified Modeling Language (UML) was also developed in the 1990s as a standard way to visualize the design of object-oriented software
Java UML Enterprise "Write Once, Run Anywhere"

Behavior These are the verbs of the software world.

Refers to the actions, operations, or transformations that the software performs. We often represent behaviors as:

  • Functions/Procedures/Methods: Named blocks of code that perform specific tasks. They can take input (data) and produce output
  • Algorithms: A specific sequence of steps or rules that a behavior follows to achieve a particular result
  • Processes/Threads: In concurrent programming, behaviors can be executed concurrently as separate processes or threads
  • Event Handlers: Behaviors triggered by specific events (e.g., a mouse click, a network request)

Data It's the nouns of the software world.

Represents the information that the software manipulates. Data can be:

  • Primitive Data Types: Basic building blocks like integers, floating-point numbers, characters, booleans
  • Data Structures: Organized collections of data, such as arrays, lists, trees, graphs, dictionaries
  • Objects: In OOP, objects combine data (attributes/fields) and the behaviors (methods) that operate on that data
  • Files/Databases: Persistent storage mechanisms that hold large amounts of data
  • Network Streams: Data flowing over a network connection

The Interaction Between Behavior & Data

The key is the interaction between behavior and data.

Software is not just about having data or having behaviors; it's about the meaningful transformations that behaviors apply to data.

This transformation is what solves problems, automates tasks, and provides value.

OOP is all about behaviors!

Each object has its behavior

A behavior == a set of parametrized operations

An object is recognized by its behavior, not its data.

Discipline of OOP

  • Data must be used in a small scope called object
  • Only the object has the direct access to its (internal) data encapsulation
  • The outside interacts with the object A by using the behavior exposed/provided by A
  • Objects know about each other through behaviors, and make use of those behaviors by interacting/communicating abstraction
OOP imposes discipline on how we transfer data indirectly — through behaviors, not direct access.

The 4 Characteristics of Behaviors

Abstraction

The behavior of an object is its abstraction: the most essential thing, the most impressive thing, stable / unlikely to change.

Interface

Encapsulation

The behavior of an object hides its (internal) details. Bundles data and methods together.

Data Hiding

Polymorphism

Behaviors are polymorphic. Same interface, different implementations across objects.

Many Forms

Inheritance

Behaviors are inheritable. Child classes reuse and extend parent behaviors.

Car Class: Python Code

import random
import math

class Car:
    def __init__(self, make, model, year):
        self._make = make
        self._model = model
        self._year = year
        self._fuel_level = 100
        self._current_speed = 0
        self._odometer = 0

    def drive(self, distance):
        fuel_consumed = self._calculate_fuel_consumption(distance)
        if fuel_consumed <= self._fuel_level:
            self._fuel_level -= fuel_consumed
            self._odometer += distance
            print(f"Drove {distance} miles. Fuel remaining: {self._fuel_level:.2f}"
                  f"Total mileage: {self._odometer}")
        else:
            print("Not enough fuel to cover the distance.")

Car: Private Helper Methods

def _calculate_fuel_consumption(self, distance):
    base_efficiency = 30
    optimal_speed = 45
    weight_factor = 0.005
    vehicle_weight = 3000

    avg_speed = self._calculate_average_speed()
    efficiency_factor = self._calculate_efficiency_factor(avg_speed)
    terrain_factor = self._calculate_terrain_factor()
    wind_resistance = self._calculate_wind_resistance(avg_speed)

    fuel_consumed = (distance / (base_efficiency * efficiency_factor * terrain_factor))
    return fuel_consumed

def _calculate_average_speed(self):
    speed_samples = [random.uniform(30, 70) for _ in range(10)]
    return sum(speed_samples) / len(speed_samples)

def _calculate_efficiency_factor(self, avg_speed):
    optimal_speed = 45
    return 1 - 0.05 * abs(avg_speed - optimal_speed)
The underscore prefix _ signals these methods are internal — not meant to be called from outside the class.

Television Class: Python Code

import random
import time

class Television:

    def __init__(self):
        self._is_on = False
        self._channel = 1
        self._volume = 50

    def toggle_power(self):
        """Turns the TV on or off."""
        self._is_on = not self._is_on

    def adjust_volume(self, volume):
        """Adjusts the volume level."""
        if self._is_on:
            self._volume = max(0, min(100, self._volume + volume))

    def change_channel(self, channel_number):
        """Changes the channel to the specified number."""
        if self._is_on and 1 <= channel_number <= 999:
            self._channel = channel_number

What is Encapsulation?

Encapsulation is the bundling of data and methods that operate on that data within a single unit (class), and restricting direct access to some of an object's components.

Key idea: An object's internal state should only be modified through its own methods — never directly from outside.
Data Bundling Access Control Information Hiding

Public vs Protected vs Private

Access levels in Python:

Level Syntax Convention Example
Public self.name Accessible everywhere obj.name
Protected self._name Convention: internal use only obj._name (discouraged)
Private self.__name Name mangling by Python obj._ClassName__name
Python doesn't enforce access control like Java or C++. The underscore convention is a gentleman's agreement among developers.

The @property Decorator

Python's way to create controlled access to attributes: 

class BankAccount:
    def __init__(self, balance):
        self._balance = balance

    @property
    def balance(self):
        return self._balance

    @balance.setter
    def balance(self, amount):
        if amount >= 0:
            self._balance = amount
        else:
            raise ValueError("Balance cannot be negative")
  
account = BankAccount(1000)
print(account.balance)      # 1000 (uses getter)
account.balance = 500       # uses setter with validation
account.balance = -100      # raises ValueError!

Getters & Setters: Why?

Without (Direct Access)

account = BankAccount(1000)

# Anyone can set anything!
account.balance = -9999
account.balance = "hello"
account.balance = None

No validation, no control, no safety.

With (Properties)

account = BankAccount(1000)

# Controlled through setter
account.balance = 500   # OK
account.balance = -100  # Error!
# Validation happens
# automatically

Validation, logging, computed values — all possible.

Encapsulation Benefits

Data Protection

Prevent invalid states. No one can set balance = -9999 if your setter validates.

Flexibility

Change internal implementation without breaking external code. Rename _balance to _amount? No problem.

Debugging

Centralized access points. If balance changes unexpectedly, check the setter — that's the only way in.

Maintainability

Clear interface means other developers know exactly how to interact with your objects.

Module-Level Encapsulation

Encapsulation in Python also applies at the module level:

Private Functions

# mymodule.py

def public_function():
    return "I'm public"

def _private_helper():
    return "I'm internal"

Functions prefixed with _ are considered internal to the module.

__all__ Exports

# mymodule.py

__all__ = ['public_function']

def public_function():
    return "Exported"

def _secret():
    return "Hidden"

__all__ controls what from module import * exports.

Best Practices

  • Use properties instead of direct attribute access for controlled data
  • Validate in setters — never trust external input
  • Keep the interface minimal — expose only what's necessary
  • Document the public API clearly
  • Use single underscore _ for protected, double underscore __ sparingly for name mangling
Most important rule: Start with everything private. Only make things public when there's a clear reason to do so. It's easier to loosen restrictions than to tighten them later.

Recap: Key Takeaways

Encapsulation

Bundles data + methods into a single unit (class). Internal state is protected from outside interference.

Access Control

Public, protected, private — Python uses naming conventions (_, __) to signal access levels.

@property

Provides a clean interface for attribute access with built-in validation through getters and setters.

Convention over Enforcement

Python trusts developers. Underscore conventions signal intent rather than enforcing strict access control.
